## 1.  발표 주제 선정 이유 (Overview)

### 1.1. 초기 기획: UX 비교 분석
프로젝트의 시작은 단순했습니다. 국내 배달 앱 시장을 양분하고 있는 **'배달의민족'**과 **'쿠팡이츠'**의 사용자 리뷰를 수집하여, 실제 소비자들이 느끼는 장단점을 데이터로 비교 분석하고자 했습니다.

### 1.2. 뜻밖의 발견 (Anomaly Detection)
데이터 수집 중 이상 징후를 발견했습니다. 쿠팡이츠는 1점부터 5점까지 점수가 고르게 분포된 반면, **배달의민족 데이터는 수집된 15,000건이 100% '5점 만점'**이었습니다. 실제 커뮤니티의 불만 여론과는 완전히 상반된 데이터였습니다.

### 1.3. 프로젝트 방향 전환: "가짜 5점을 찾아라"
**"이 5점들은 진짜일까?"**
단순 비교를 넘어, 이 비정상적인 평점의 원인을 파헤치기로 했습니다. 본 프로젝트는 **별점(Quantitative)과 텍스트(Qualitative)의 괴리**를 분석하여 소비자의 숨겨진 진짜 목소리를 찾아 배달앱의 보완점을 도출해 내는 것으로 주제를 심화했습니다.

## 2.  데이터 수집 및 기술 (Data Collection & Methodology)

### 2.1. 기술 스택 및 수집 도구
총 **22,705건**의 유효 리뷰 데이터를 확보했습니다.

| 구분 | **배달의민족 (Baemin)** | **쿠팡이츠 (Coupang Eats)** |
| :--- | :--- | :--- |
| **수집 도구** | **Selenium Web Driver** | **Google-Play-Scraper API** |
| **선정 이유** | 구글의 봇 차단 정책 및 동적 페이지 로딩 문제 (무한 스크롤 이슈)를 해결하기 위해 브라우저 제어 방식(Selenium) 채택 | 라이브러리를 활용하여 대량의 데이터를 고속으로 수집 |
| **데이터 양** | **15,000건** (100% 5점 데이터) | **7,705건** (1~5점 고른 분포) |

### 2.2. 데이터 전처리 및 품질 관리 (Data Quality Control)
분석 결과의 신뢰도를 높이기 위해 **2단계 노이즈 필터링**을 수행했습니다.
1.  **Low-Quality 제거:** "굿", "좋음", "ㅋㅋㅋㅋ" 등 분석 가치가 없는 **10글자 미만**의 단답형 리뷰는 수집 단계에서 제외했습니다.
2.  **노이즈 정제 (Regex):** 텍스트 분석 시 방해가 되는 자음/모음 남발(`ㅠㅠ`, `ㅎㅎ`)과 특수문자를 정규표현식으로 제거하고, **유의미한 한글 단어**만을 추출했습니다.

## 3.  분석 방법론 (Methodology: AI & NLP)

단순 통계 분석을 넘어, 최신 NLP(자연어 처리) 기법을 도입하여 분석의 정밀도를 높였습니다.

### 3.1. 데이터 라벨링 (Data Labeling)
* **목표:** 지도 학습(Supervised Learning)을 위한 학습 데이터셋 구축 (전체의 약 11%).
* **규모:** 무작위 추출한 **2,500건**에 대해 직접 라벨링 수행.
* **라벨링 도구:** Google Gemini Pro를 활용한 1차 분류 후 인간 검수(Human-in-the-loop).
* **Class 정의:**
    * `Class 0 (True 5-Star)`: 순수 만족 리뷰 ("맛있고 빨라요")
    * `Class 1 (Fake 5-Star)`: 평점은 5점이나 내용은 불만인 리뷰 ("오류 고치라고 5점 줌")

### 3.2. 모델링 프로세스 (Modeling)
#### ① 문장 분류: KoELECTRA Fine-tuning
* **모델:** **monologg/koelectra-base-v3-discriminator**
* **이유:** 한국어의 문맥과 구어체(리뷰) 이해도가 높은 사전학습 모델 사용.
* **과정:** 라벨링된 2,500건으로 파인튜닝(Fine-tuning) 후, 나머지 20,000건의 감성을 예측 분류.

#### ② 토픽 모델링: BERTopic / LDA
* **목표:** '가짜 5점(불만)'으로 분류된 리뷰들이 구체적으로 무엇을 이야기하는지 파악.
* **기법:** 문맥 기반의 클러스터링을 통해 주요 불만 토픽(Topic) 추출.
    * *Topic 1:* 시스템 안정성 (업데이트, 렉, 발열)
    * *Topic 2:* 고객 응대 (고객센터, 상담원)

## 5.  배민이 5점이 많은 이유: 알고리즘과 사용자 심리의 공모

왜 배민은 5점뿐일까요? 심층 분석 결과, 이는 **플랫폼의 노출 로직**과 **사용자의 전략적 행동**이 맞물린 결과였습니다.

### 5.1. 플랫폼의 '관련도순' 알고리즘 편향
구글 플레이 스토어의 **'관련도순' 정렬**은 **"평점이 높고(5점), 텍스트 길이가 긴 리뷰"**를 가장 가치 있는 정보로 판단하여 최상단에 노출시킵니다. 이로 인해 크롤링 시 5점 리뷰만 집중적으로 수집되었습니다.

### 5.2. 사용자의 '트로이 목마' 전략 (The "Trojan Horse" Strategy)
사용자들은 자신의 불만이 묻히는 것을 원하지 않습니다. 낮은 평점을 주면 리뷰가 하단으로 밀려나거나 운영진이 보지 않을 것이라 생각하기 때문에, **'가짜 5점'**을 부여하여 상단에 노출시키고 실제 내용에는 강한 비판을 적는 **지능적인 시위(Protest)** 행동을 보입니다.

####  [실제 데이터 예시] 5점을 준 사용자의 진짜 속마음
> **Case 1 (⭐⭐⭐⭐⭐):** "리뷰는 느낀 점을 적는 건데 별점 낮게 줬다고 업주가 신고해서 차단시키네요? 고객센터 연결도 안 되고 진짜 **짜증**납니다. 보라고 5점 줍니다."
>
> **Case 2 (⭐⭐⭐⭐⭐):** "첫 주문 쿠폰 준대서 가입했는데 안 써집니다. 장난하나요? **업데이트** 이후로 렉도 심하고 바로 **삭제**합니다."
>
> **Case 3 (⭐⭐⭐⭐⭐):** "제발 **오류** 좀 고치세요. 돈 내고 쓰는 배민클럽인데 혜택은커녕 접속도 안 되면 어쩌라는 겁니까?"

## 7.  쿠팡이츠 데이터 분석 (Coupang Eats Analysis)

쿠팡이츠는 **"속도는 최고지만, 디테일한 품질 관리가 필요하다"**는 평가를 받았습니다.

### 7.1. 긍정적 요인 - 진짜 만족
쿠팡이츠의 5점 리뷰는 부정적 내용이 5.3%에 불과한 '진짜 만족'이었습니다.

* **인사이트:** 복잡한 기능 없이 **"주문 → 도착"** 과정이 간결하고 빠르다는 점이 5점 리뷰의 주된 이유였습니다. 특히 와우 멤버십 연계 혜택에 대한 충성도가 높습니다.

* **인사이트:** 배달 속도는 빠르지만 **라이더의 배달 품질(길 못 찾음, 공동현관 문제, 음식 파손)**에 대한 불만이 많습니다. 또한 배달비나 최소주문금액이 부담스럽다는 의견이 존재합니다.

## 8.  종합 제언: 강점은 살리고 약점은 보완하라 (Strategic Proposal)

데이터 분석 결과를 바탕으로, 각 플랫폼이 시장 우위를 점하기 위한 **'강점 강화(Maintain)'** 및 **'약점 보완(Improve)'** 전략을 제언합니다.

###  배달의민족 전략: "압도적 1위의 품격 갖추기"

| 구분 | 내용 |
| :--- | :--- |
| **강점 유지 (Maintain)** | **"압도적 혜택"**<br>타사 대비 월등한 가맹점 수와 끊임없는 쿠폰 이벤트는 배민의 가장 강력한 무기입니다. 현재의 프로모션 전략을 유지하여 사용자 락인(Lock-in) 효과를 지속해야 합니다. |
| **개선 과제 (Improve)** | **"기술적 부채 해결 (Tech Stability)"**<br>1. **앱 경량화:** 화려한 기능보다 '켜지는 앱'이 우선입니다. 업데이트 후 발생하는 렉과 발열을 잡아야 합니다.<br>2. **Human Touch:** 챗봇 반복이 아닌 '직통 상담 채널'을 열어 답답한 고객 경험(CX)을 개선해야 합니다. |

###  쿠팡이츠 전략: "속도를 넘어 품질로"

| 구분 | 내용 |
| :--- | :--- |
| **강점 유지 (Maintain)** | **"본질적 속도"**<br>복잡한 기능 없이 '주문-도착'에 집중한 직관적인 UI와 압도적인 배달 속도는 쿠팡이츠만의 정체성입니다. 이 심플함(Simplicity)을 잃지 말아야 합니다. |
| **개선 과제 (Improve)** | **"디테일 품질 관리 (Quality Control)"**<br>1. **라이더 교육:** 속도는 빠르지만 '음식이 식거나 쏟아지는' 경우가 있습니다. 배달통 보온 및 고객 응대 교육을 강화해야 합니다.<br>2. **정보 가독성:** 배민에 비해 가게 공지나 리뷰 보기가 불편합니다. 정보 전달력을 높이는 UI 개선이 필요합니다. |

## 9.  트러블 슈팅 및 난관 해결 (Troubleshooting)

프로젝트 진행 중 직면했던 **두 가지 결정적인 난관**과 이를 극복한 과정을 정리했습니다.

###  난관 1: "Selenium의 늪" (크롤링 기술적 문제)
* **문제 상황 (Problem):** 처음에는 일반적인 크롤링 라이브러리(`Scraper`)를 사용하려 했으나, 배달의민족 페이지 구조상 데이터 접근이 제한되는 문제가 발생했습니다. Selenium으로 전환했으나, 이번에는 **무한 스크롤 중단 및 요소 로딩 지연** 문제로 인해 2~3천 건 수집 후 코드가 멈추는 현상이 반복되었습니다.
* **해결 (Solution):**
    * 단순 스크롤(`scrollTo`) 방식 대신, **'더보기 버튼'을 명시적으로 찾아 클릭(Click)**하고, `Explicit Wait`(요소가 뜰 때까지 대기) 로직을 추가하여 브라우저의 로딩 속도에 맞춰 동적으로 작동하도록 개선했습니다.
    * 이 과정을 통해 15,000건의 데이터를 끊김 없이 안정적으로 확보하는 데 성공했습니다.

###  난관 2: "데이터 편향의 딜레마" (분석적 문제)
* **문제 상황 (Problem):** 어렵게 수집한 15,000건의 배달의민족 데이터를 열어본 순간 당황했습니다. **모든 리뷰의 평점이 '5점'**이었기 때문입니다. 처음에는 크롤링 로직이 잘못되어 5점만 가져온 것이라 생각하고 코드 수정에 많은 시간을 쏟았습니다.
* **해결 (Solution):**
    * 수집된 데이터를 직접 구글 플레이 스토어 화면과 대조하며 검증한 결과, 코드는 정상이며 **구글의 상위 노출 알고리즘 자체가 5점을 우선시한다**는 사실을 깨달았습니다.
    * 이를 단순한 오류로 치부하지 않고, **"왜 사용자들은 불만이 있으면서도 5점을 줬을까?"**라는 질문으로 전환하여 **'트로이 목마 전략(가짜 5점)'**이라는 핵심 인사이트를 도출해냈습니다.
